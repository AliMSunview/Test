Managing Task Dependencies

In addition to ordering tasks, the task owner can also enforce the order in which tasks are worked on. With task dependencies, users assigned to a task cannot begin the task until all preceding tasks have been completed. Tasks can be moved into different orders, where lower numbers indicate higher priority tasks. For example, a user cannot accept a task in order 3 until the tasks in orders 1 and 2 have already been completed. Tasks that are grouped in the same order can be worked on simultaneously.

For example, imagine that a ticket has been created to provision a new server. The ticket contains the following tasks:

  • Procure hardware

  • Install and configure operating system

  • Install application software

  • Set up security

Installation and configuration cannot be performed until the server hardware has been procured. However, application installation and security setup can be performed simultaneously since one does not need to be completed before the other can begin. After reordering and grouping each of these tasks, the final task list may look similar to the following:

  1. Procure hardware

  2. Install and configure operating system

  3. Install application software

  1. Set up security

Active vs. Inactive Tasks

A task will automatically toggle between active and inactive depending on its order in the task queue. An active task can be worked on at any time by its implementer, whereas an inactive task cannot be started until all of the active tasks before it have been closed. A task will automatically switch from inactive to active when all of its preceding tasks have been completed.

Moving an active task to an inactive group sets the task as inactive. This occurs immediately and prevents the implementer from working on the task until it becomes active again. If you move an inactive task to a group of active tasks, then the task becomes active and must be completed before the next group of tasks is marked as active.

Note: Reopening a closed task does not affect the dependency chain. Any tasks that are currently active will remain active.

Enabling Task Dependency

The task dependency feature is enabled on a per-module basis by a ChangeGear administrator. However, additional functionality is available to staff users through the ChangeGear Web client.

Depending on your ChangeGear configuration, you may also be able to manually enable or disable the task dependency feature for individual tasks. Enabling it prevents that task from changing its current state until all preceding tasks have been completed. Disabling it means that the task can be accepted, worked on, and closed at any point in the ticket's life cycle.

To manage task dependencies for individual tasks:

  1. Open the ticket that you wish to modify.

  2. Navigate to the Tasks (or Implementation) tab.

  3. Open the task that you wish to manage dependencies for.

  4. Check the Enforce task dependency checkbox to enforce dependencies for the task

    - OR -

    Uncheck the Enforce task dependency checkbox to allow the task's implementer to start the task at any time.

    Note: If you do not see the Enforce task dependency check box, please contact your ChangeGear administrator.

  5. Click the Save button to save the task.
